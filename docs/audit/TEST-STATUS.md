# Audit RED tests — run status

**Date:** 2026-07-21
**Status (updated 2026-07-21):** all three CRITICAL fixes are **applied and build-verified**
(`anchor build` passes locally and in CI on Linux). The RED tests were **converted from
documents-the-bug to regression (asserts-the-guard-reverts)**. CI compiles all programs; the full
runtime `anchor test` suite is **not** run in CI — see "CI status" below for why.

## CI status (GitHub Actions, `.github/workflows/anchor-test.yml`)

CI runs on Linux (the local Solana test validator wedges on Windows). It is a **build-only gate**:
it compiles all 12 programs and fails the check if compilation breaks. It does **not** run the
runtime `anchor test` suite, for two independent reasons discovered while wiring it up:

1. **Anchor 0.31.1's IDL build is incompatible with the dependency tree.** It installs and uses its
   own rust-1.79 `solana` rustup toolchain (uninstalling any override), which cannot parse the
   `edition2024` crates the tree pulls in (`crypto-common`, `blake3`, `block-buffer`, `toml_edit`).
   The SBF program build itself works fine with platform-tools **v1.52 (rust 1.89)** — so CI builds
   with `anchor build --no-idl -- --tools-version v1.52`.
2. **The runtime suite needs stable program IDs.** `anchor test` deploys each program at the address
   in its `target/deploy/*-keypair.json`, and the program asserts that address equals its
   `declare_id!` at runtime (`DeclaredProgramIdMismatch`, error 4100). A fresh CI build generates
   **random** program keypairs that do not match `declare_id!`, so every test's `before` hook fails.
   The standard fix is to commit the program keypairs to pin the IDs — declined per the project rule
   "never commit private keys". (Note this also means `anchor test` was never green in this repo; it
   had never run, because the Windows validator always wedged.)

**To run the full suite** (locally on Linux/macOS, or in CI if the program keypairs are committed):
`anchor build --no-idl -- --tools-version v1.52 && anchor test --skip-build` with the IDL/TS types
staged from `idl/` and `types/` (committed, generated by the local build).

## Why they were not run green here

`solana-test-validator` was unstable on this Windows machine across several attempts on 2026-07-21:

1. First run: `Failed to create ledger ... Acceso denegado (os error 5)` — a locked/leftover `.anchor/test-ledger`. Fixed by removing the ledger.
2. Second run: `Test validator does not look started` — genesis-unpack + 12-program deploy exceeded the default startup wait. Mitigated by adding `[test] startup_wait = 120000` to `Anchor.toml`.
3. Third/fourth runs: validator started ("Running test suite" printed) but then **wedged for 13–40 min** with no mocha output and the `solana-test-validator` process gone — the validator died mid-run. Reproduced twice after a clean ledger.

This is an **environment issue, not a test-logic issue**. The underlying bugs are independently **verified by manual code review** (see `2026-07-21-perp-vault-findings.md` CRITICAL-1 and `2026-07-21-perp-engine-findings.md` HIGH), reading the exact deduct/credit + serialization-order and the settlement-skip branches.

## To run these green

Run on a **non-Windows environment** (Linux/macOS or CI), where `solana-test-validator` is reliable:

```bash
anchor test          # builds, starts validator, runs tests/*.ts in order
# or, against a manually-started validator:
solana-test-validator &   # wait until healthy
anchor test --skip-local-validator
```

Expected: the full suite passes, including the two RED tests above (which pass while the bugs are present).
